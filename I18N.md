# i18n & SEO pipeline (ARY-401)

How the marketing site does **crawlable, localized, SEO-correct pages**. English
ships first; **ES / FR / DE** (CMO ARY-399 §3, CEO-approved Phase-1 set) layer in
as a content-only operation — no routing code changes.

## TL;DR for adding a language to a page

1. Translate the page body (CMO owns copy; UXDesigner owns layout).
2. Create the localized file at `src/pages/<locale>/<page>.astro` (English lives
   at the root un-prefixed, e.g. `src/pages/security.astro` → `/security/`;
   Spanish is `src/pages/es/security.astro` → `/es/security/`). Render through
   `<Base path="/security/" locale="es" …>`.
3. In `src/config/i18n.ts`, add the locale to that path's `PAGE_LOCALES` entry
   (e.g. `'/security/': ['en', 'es']`).
4. `npm run build`. hreflang pairs, the self-canonical, `<html lang>`, and the
   localized sitemap (`xhtml:link` alternates) all update automatically.

That's it. Routing, `hreflang`, canonicals, and the sitemap are derived — never
hand-maintained.

## The moving parts

| Concern | Owner file | What it does |
|---|---|---|
| Locales + hreflang correctness | `src/config/i18n.ts` | Declares the Phase-1 locales, BCP-47 hreflang codes, and **`PAGE_LOCALES`** — the per-path list of locales that actually have a live page. hreflang/sitemap only ever link locales listed here. |
| Routing | `astro.config.mjs` | Astro `i18n` config. `prefixDefaultLocale: false` keeps English URLs clean (`/security/`, not `/en/security/`). |
| Localized sitemap | `astro.config.mjs` (`@astrojs/sitemap` `i18n`) | Emits `xhtml:link rel="alternate"` pairs per page across its built locales. |
| Per-page/per-locale `<title>`, meta, canonical, OG, hreflang | `src/layouts/Base.astro` → `src/components/Seo.astro` | One `<Seo/>` per page. Pass `locale`, `title`, `description`, `path`. |
| Structured data (JSON-LD) | `src/config/structured-data.ts` + `Seo.astro` | Organization + WebSite site-wide; SoftwareApplication auto on home; `faq` / `breadcrumbs` props opt pages into FAQPage / BreadcrumbList; BlogPosting on articles. |
| UI chrome strings | `src/i18n/ui.ts` | `useTranslations(locale)` → `t(key)`, English-fallback. Nav, CTAs, footer. |

## Why `PAGE_LOCALES` is the keystone (the #1 multilingual-SEO rule)

Per ARY-399 §2e, **hreflang/sitemap must only point at pages that exist.** A
`hreflang="es"` link to a `/es/...` URL that 404s is the single most common
multilingual SEO bug. `PAGE_LOCALES` lists, per path, the locales whose page is
actually published. Until a translation lands, a page stays English-only and
emits a clean `en` + `x-default` — never a broken alternate. Verified
end-to-end: adding an `es` page makes `/` and `/es/` cross-link bidirectionally
in both `<head>` and the sitemap; removing it reverts to `en` + `x-default`.

## SEO checklist status (ARY-399 §2e)

- [x] Crawlable static pages (not the SPA) — Astro static output
- [x] Unique `<title>` / meta per page **and per locale** — `Base`/`Seo`
- [x] Canonical URLs (locale-aware, self-canonical) — `Base`
- [x] `hreflang` for every live localized pair + `x-default` — `i18n.ts`/`Seo`

- [x] JSON-LD: Organization, SoftwareApplication (FinanceApplication, free),
      FAQPage, BreadcrumbList, Article — `structured-data.ts`
- [x] Fast LCP — static, light Calm design, no client JS framework
- [ ] One `<h1>` per page — enforced by content/layout (UXDesigner); infra ready

## Translation tooling

`src/i18n/ui.ts` overlays are `Partial` per locale with English fallback, so a
half-translated locale never ships blanks. `untranslatedKeys(locale)` lists what
still needs translating (CI/QA hook). Drafting tool decision (DeepL Pro vs MT) is
the CEO/board budget call in ARY-399 §5 — the pipeline is tool-agnostic.
