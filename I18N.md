# i18n & SEO pipeline (ARY-401, full-locale machine translation ARY-427)

How the marketing site does **crawlable, localized, SEO-correct pages**. Every
Phase-1 page is published in the **full set of locales the Vaulto app supports**
(16: en, es, fr, pt, de, it, nl, zh, ja, ar, hi, ru, pl, tr, ko, sv — enumerated
to match the app's i18n catalog). English is the canonical source; every other
locale is a **$0 machine translation** generated via the OpenAI API by a
committed, repeatable script (board directive ARY-425).

## Architecture (content-only localization)

- **English source of truth** lives as JSON under `src/i18n/content/<group>/en.json`,
  one group per surface: `chrome` (nav/footer/CTAs), `home`, `how-it-works`,
  `features`, `planning`, `security`, `privacy`, `faq`. Only translatable *text*
  is in JSON — emoji/icons, URLs, hrefs, and email links stay in the components.
- **Translations** are sibling files `src/i18n/content/<group>/<locale>.json`,
  generated by `scripts/translate.mjs`. `getContent(group, locale)` deep-merges
  the locale overlay over English, so any missing/blank string falls back to
  English — a partially-translated locale never ships a hole.
- **One page component per page** (`src/components/pages/<Name>Page.astro`, plus
  `HomePage.astro`) takes a `locale` prop and renders from `getContent`.
- **Routing**: the English page is un-prefixed at `src/pages/<page>.astro`; a
  single dynamic route `src/pages/[locale]/<page>.astro` (`getStaticPaths` over
  `NON_DEFAULT_LOCALES`) emits every other locale (`/fr/security/`, `/ja/faq/`, …).
  Arabic gets `<html dir="rtl">` automatically (`dirFor` in `config/i18n.ts`).

## Re-translating after an English copy change ($0, repeatable)

1. Edit the English JSON (`src/i18n/content/<group>/en.json`).
2. Pull the OpenAI key at run time (never commit it) and run the script:
   ```sh
   # key lives on the Render API service env (AI_INTEGRATIONS_OPENAI_API_KEY);
   # export OPENAI_API_KEY / OPENAI_BASE_URL, then:
   node scripts/translate.mjs --group=<group> --force   # re-translate one group
   node scripts/translate.mjs                            # fill only missing files
   ```
   The script preserves brand "Vaulto", HTML tags, emoji, placeholders, and the
   security/legal claim semantics (glossary + guardrails baked into the prompt),
   and structurally validates each result before writing.
3. `npm run build`. hreflang, self-canonicals, `<html lang/dir>`, and the
   localized sitemap (`xhtml:link` alternates) all update automatically.

## Adding a NEW locale

Add the code to `LOCALES` (+ `LOCALE_NAMES`, `HREFLANG`, and `RTL_LOCALES` if
right-to-left) in `src/config/i18n.ts`, run the script, build. No per-page or
routing edits — the dynamic routes and `PAGE_LOCALES` derive from the config.

## The moving parts

| Concern | Owner file | What it does |
|---|---|---|
| Locales + hreflang correctness | `src/config/i18n.ts` | Declares the Phase-1 locales, BCP-47 hreflang codes, and **`PAGE_LOCALES`** — the per-path list of locales that actually have a live page. hreflang/sitemap only ever link locales listed here. |
| Routing | `astro.config.mjs` | Astro `i18n` config. `prefixDefaultLocale: false` keeps English URLs clean (`/security/`, not `/en/security/`). |
| Localized sitemap | `astro.config.mjs` (`@astrojs/sitemap` `i18n`) | Emits `xhtml:link rel="alternate"` pairs per page across its built locales. |
| Per-page/per-locale `<title>`, meta, canonical, OG, hreflang | `src/layouts/Base.astro` → `src/components/Seo.astro` | One `<Seo/>` per page. Pass `locale`, `title`, `description`, `path`. |
| Structured data (JSON-LD) | `src/config/structured-data.ts` + `Seo.astro` | Organization + WebSite site-wide; SoftwareApplication auto on home; `faq` / `breadcrumbs` props opt pages into FAQPage / BreadcrumbList; BlogPosting on articles. |
| UI chrome strings | `src/i18n/ui.ts` | `useTranslations(locale)` → `t(key)`, English-fallback. Nav, CTAs, footer. |

## Why `PAGE_LOCALES` is the keystone (the #1 multilingual-SEO rule)

Per ARY-399 §2e, **hreflang/sitemap must only point at pages that exist.** A
`hreflang="es"` link to a `/es/...` URL that 404s is the single most common
multilingual SEO bug. `PAGE_LOCALES` lists, per path, the locales whose page is
actually published. Until a translation lands, a page stays English-only and
emits a clean `en` + `x-default` — never a broken alternate. Verified
end-to-end: adding an `es` page makes `/` and `/es/` cross-link bidirectionally
in both `<head>` and the sitemap; removing it reverts to `en` + `x-default`.

## Translation tooling

`src/i18n/ui.ts` overlays are `Partial` per locale with English fallback, so a
half-translated locale never ships blanks. `untranslatedKeys(locale)` lists what
still needs translating (CI/QA hook). Drafting tool decision (DeepL Pro vs MT) is
the CEO/board budget call in ARY-399 §5 — the pipeline is tool-agnostic.
